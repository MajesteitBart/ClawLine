# Product Context

## Users
- Primary user: Bart as the operator of his own OpenClaw system.
- Secondary future users: trusted collaborators with narrower access and less debug depth.
- Device role: the phone is both a mobile client and, later, a potential node companion.

## Core Flows
- Connect securely to an OpenClaw gateway from mobile.
- Browse and switch across active sessions.
- Read and continue chat in a clean session-first interface.
- Inspect live tool activity, approvals, failures, and operator-relevant run state.
- Check gateway, node, channel, skill, and cron health from the phone.
- Use voice, attachments, and mobile ergonomics where typing is slower or less practical.

## Constraints
- The app must feel more native and organized than Telegram, but more operational than ChatGPT.
- Gateway remains the source of truth.
- Clean chat and deep debug/operator views should be adjacent, not mixed together by default.
- Alpha scope must stay tight: core chat, activity, and console value first, deeper node/device ambition later.
